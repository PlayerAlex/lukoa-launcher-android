# 露科亚启动器 UI / UX 重构框架

这份文档交给 UI / UX 设计师使用。它定义产品结构、状态、功能边界和设计交付要求，不规定配色、字体、图标、插画、动效或具体美术风格。

本次不是给旧界面换皮。当前“文档 / 版本 / 启动 / 备份 / 设置”和设置内部大量横向分区过于零散，新设计应重新组织信息层级。

## 一、重构目标

用户进入 App 后应很快得到四个答案：

1. 当前操作的是哪个酒馆实例？
2. 酒馆现在能不能用、是否正在运行？
3. 最常用操作在哪里？
4. 出错后下一步该做什么？

界面面向不熟悉 Termux、Linux、Git、npm 和 Android 权限的新手。专业信息可以保留，但必须分层展示。

## 二、建议的新信息架构

一级入口压缩为四个：

```text
露科亚启动器
├─ 首页
├─ 酒馆
│  ├─ 实例
│  ├─ 版本
│  ├─ 用户
│  └─ 维护
├─ 数据
│  ├─ 备份库
│  ├─ 自动备份
│  └─ 导入与导出
└─ 设置
   ├─ 环境与权限
   ├─ 网络与镜像
   ├─ 启动器更新
   ├─ Termux 唤醒
   └─ 诊断与日志
```

“帮助 / 文档”不再占用一级导航，建议放在首页右上角、设置页顶部或全局帮助入口。

### 为什么这样分

- 首页只负责“状态与立即操作”。
- 酒馆页负责“管理当前 SillyTavern”。
- 数据页只负责“备份、恢复和文件流转”。
- 设置页负责“运行环境和启动器自身”。
- SillyTavern 版本更新与启动器 APK 更新必须分开。

## 三、全局页面框架

所有页面共享一个清楚的当前实例入口。

```text
顶部栏
├─ 页面标题
├─ 当前实例
└─ 帮助 / 更多

页面内容
├─ 状态摘要
├─ 当前最重要操作
├─ 核心内容
└─ 次要工具 / 高级选项

底部导航
├─ 首页
├─ 酒馆
├─ 数据
└─ 设置
```

当前实例至少显示：实例名称、目录短名、端口。危险操作确认中必须显示完整路径。

## 四、首页

首页只保留高频信息与操作，不展示复杂设置。

### 首屏内容

- 当前实例
- 酒馆状态：未安装、未运行、启动中、运行中、异常
- 主操作按钮
- 打开酒馆网页
- 一句话问题摘要
- 当前后台任务

### 次级内容

- 唤醒 Termux
- 最近一次备份
- 当前 SillyTavern 版本
- 快速体检
- 最近操作反馈

### 主按钮规则

| 状态 | 主按钮 |
|---|---|
| 未安装 | 安装酒馆 |
| 已安装、未运行 | 启动酒馆 |
| 启动中 | 停止酒馆 |
| 运行中 | 停止酒馆 |
| 全局操作中 | 禁用并显示任务名称与耗时 |

启动中必须保留停止入口。

## 五、酒馆页

酒馆页采用二级导航或列表入口，不建议把所有功能同时铺在一个长页面。

### 1. 实例

- 当前实例概览
- 切换实例
- 新增实例
- 删除实例
- 酒馆目录
- 端口
- 传统目录、托管目录、自定义目录迁移

实例是一整套独立酒馆。路径不能被界面自动擅自修改。

### 2. 版本

- 自动读取当前版本
- 当前分支、提交、Git 源
- 官方稳定版
- 测试分支
- 安装、更新、回退
- 自动安全备份结果

更新或回退确认必须展示当前实例、当前版本、目标版本、Git 源和安全备份状态。

### 3. 用户

用户属于当前实例，不是启动器实例。

- 登录标识
- 显示名称
- 管理员状态
- 启用状态
- 数据目录状态
- 数据目录大小
- 新增用户
- 删除账户但保留目录

登录标识是登录时使用的英文短名，也是 `data/登录标识` 目录名，不是显示昵称。SillyTavern 官方没有提供修改登录标识的接口，因此不提供重命名。

### 4. 维护

- 一键体检
- 修复 npm 依赖
- 重置网页主题
- Node.js 内存上限
- 聊天记录上传限制
- 强制清理残留进程

每个工具需要单独说明：适用场景、修改对象、是否备份、失败能否恢复、是否需要停止或重启酒馆。

## 六、数据页

数据页围绕“避免数据丢失”设计。

### 备份库

- 手动备份
- 自动备份
- 名称、时间、大小、所属实例
- 应用、复制、导出、重命名、删除

### 自动备份

- 开关
- 间隔
- 保留数量
- 下次执行时间
- 最近执行结果

### 恢复流程

```text
选择备份
→ 读取恢复预览
→ 展示备份内容与当前实例
→ 要求停止酒馆
→ 二次确认
→ 执行恢复
→ 展示旧目录和回退位置
```

“应用备份”和“删除备份”不能使用相同视觉权重。

## 七、设置页

设置页只放环境和启动器自身设置。

### 环境与权限

- Termux 是否安装
- RUN_COMMAND
- 外部调用
- 启动器后台运行
- Termux 后台常驻
- 文件访问
- 安装未知来源 APK

每个权限项包含：当前状态、用途、主操作、系统设置手动路径。

### 网络与镜像

- Git 源
- npm 源
- Termux 包源
- 可用性检测
- 恢复官方源

### 启动器更新

- 当前启动器版本
- 稳定版 / 测试版通道
- 检查更新
- Release 页面
- 安装 APK

这里管理启动器 APK，不是 SillyTavern 版本。

### Termux 唤醒

- 自动唤醒说明
- 返回启动器延迟
- 当前后台权限状态

### 诊断与日志

- 导出诊断
- 清空页面日志
- 查看酒馆日志
- 查看 App 操作反馈

清空页面日志不会删除后台诊断归档，文案必须准确。

## 八、状态设计矩阵

每个主要页面至少设计以下状态，不接受只交正常状态稿：

- 首次进入
- 无数据
- 正常
- 加载中
- 操作中
- 成功
- 普通提醒
- 可恢复错误
- 阻断错误
- 权限不足
- Termux 未安装
- 酒馆正在运行但操作要求停止
- 切换实例后等待刷新
- 长任务超时

## 九、全局忙碌状态

安装、更新、回退、备份、恢复、迁移、修复和用户修改会锁定冲突操作。

忙碌状态需要显示：

- 正在执行什么
- 已等待多久
- 为什么其他按钮不可用
- 是否需要查看 Termux
- 超时后的下一步

视觉重构不能删除重复点击保护。

## 十、危险操作分级

### 普通确认

保存设置、切换普通选项。

### 风险确认

停止、更新、回退、删除账户、删除备份、迁移目录。

### 高风险确认

应用备份、删除实例目录、强制清理进程。

风险确认必须明确展示操作对象和影响，禁止只写“确定吗”。高风险确认还需要展示完整路径、是否可恢复和恢复位置。

## 十一、组件库需求

设计师需要提供这些组件的完整状态：

- 一级导航
- 二级导航 / 功能入口
- 顶部栏
- 实例切换器
- 状态徽标
- 状态摘要卡
- 普通信息卡
- 主要、次要、危险、禁用按钮
- 输入框及错误状态
- 列表项
- 备份记录
- 用户记录
- 日志容器
- 空状态
- 加载状态
- 错误状态
- 普通确认框
- 风险确认框
- 全局忙碌层
- Snackbar / 短反馈

## 十二、设计令牌

交付物需要能映射到 Jetpack Compose：

- 语义颜色角色
- 字体层级
- 间距阶梯
- 圆角阶梯
- 描边规则
- 阴影或层级规则
- 图标尺寸
- 组件高度
- 动效时长与缓动
- 深色 / 浅色策略

颜色应按语义命名，例如 `background`、`surface`、`textPrimary`、`accent`、`success`、`warning`、`danger`、`divider`，不要只给一组无用途说明的色值。

## 十三、设计稿交付清单

1. 四个一级页面完整稿。
2. 酒馆页四个二级模块。
3. 设置页五个二级模块。
4. 首页未安装、未运行、启动中、运行中、异常状态。
5. 版本更新与回退确认。
6. 备份恢复完整流程。
7. 用户新增与删除账户确认。
8. 权限缺失和 Termux 未安装状态。
9. 全局忙碌和超时状态。
10. 空列表、加载失败和多实例切换状态。
11. 组件库和设计令牌。

设计稿需要考虑窄屏、长中文、系统字体放大和按钮文字换行。

## 十四、工程承接位置

新信息架构可以重组现有模块，但不要把大段 UI 塞回总入口。

| 设计区域 | 当前工程入口 |
|---|---|
| 一级导航 | `LauncherNavigation.kt` |
| 首页 | `LauncherLaunchSections.kt` |
| 酒馆版本 | `LauncherVersionManagementSection.kt` |
| 实例与路径 | `LauncherSettingsSection.kt`、路径状态文件 |
| 用户管理 | `TavernUserManagementSection.kt` |
| 维护工具 | `RepairToolsSection.kt`、`HealthCheckSection.kt` |
| 数据与备份 | `BackupSection.kt`、备份 DialogHost |
| 环境和设置 | `LauncherSettingsSection.kt`、`SettingsDialogs.kt` |
| 状态编排 | `LukoaLauncherScreen.kt`，只负责接线和跨模块协调 |

实现阶段应先建立新的页面容器和导航，再逐块搬迁现有 Section；每次只迁一个业务模块，并保持行为和测试不变。

## 十五、不可破坏的规则

- 默认简体中文和新手文案。
- 当前实例在危险操作前必须清楚可见。
- 默认路径规则不能改变。
- 自动读取当前 SillyTavern 版本必须保留。
- SillyTavern 更新与启动器 APK 更新必须区分。
- 备份恢复优先保护数据。
- 危险操作必须保留确认和恢复说明。
- 不允许删除全局操作锁。
- 不允许用只有图标、没有文字的方式表达关键危险操作。
- 不允许把错误详情藏到用户难以找到的位置。

## 十六、设计师可以自由决定

- 品牌视觉
- 配色
- 字体
- 图标
- 插画
- 页面布局
- 导航的视觉形式
- 卡片风格
- 动效
- 空状态表现
- 信息层级的视觉表达

只要满足信息架构、状态覆盖、安全规则和可访问性要求，就不需要延续当前 UI 的任何美术风格。
