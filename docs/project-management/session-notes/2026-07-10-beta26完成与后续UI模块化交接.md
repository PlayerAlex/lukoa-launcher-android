## 4. 当前 UI 大文件现状

当前代码统计：

- `LukoaLauncherScreen.kt`：约 4575 行。
- `LukoaLauncherScreen` 内：约 104 个 `var` 状态、152 个本地函数。
- `LauncherDialogs.kt`：约 1950 行、23 个顶层对话框函数，另有多个私有辅助组件。

`LukoaLauncherScreen.kt` 当前同时承担：

1. 页面主状态和持久化状态恢复。
2. Termux 结果归并、日志归并和运行状态推断。
3. 操作锁、PendingLauncherTask 和冷启动恢复。
4. 备份列表、创建、恢复、复制、导入、重命名和删除流程。
5. 酒馆安装、更新、回退和安全备份串联。
6. 多实例、目录迁移、托管目录删除和路径设置。
7. Termux 权限、包源、环境准备和体检。
8. GitHub 更新检查、下载和安装。
9. 约 25 个对话框的显示条件与回调编排。
10. Pager、导航和各页面 section 的最终组装。

真正风险不是行数本身，而是一次改动容易跨多个业务域影响状态、锁、PendingTask 和持久化。

## 6. 拆分总原则

1. 先做机械搬迁，再做状态架构调整；不要同一版同时移动文件、改交互、改文案和改业务。
2. 每版只拆一个业务域或一种职责，默认发布一个新的 prerelease。
3. 第一阶段保留 `LukoaLauncherScreen` 对状态的所有权，只把对话框和渲染编排移出。
4. 第二阶段才逐个引入业务域 state holder/coordinator。
5. 不要一次性创建一个包含 100 个字段的 `LauncherScreenState`，这只是把大文件换成大对象。
6. 不要一次性引入完整 ViewModel 重写；当前大量状态与 Compose、Job、请求 token、持久化回调紧密关联。
7. 每个新 state holder 只负责一个业务域，并明确谁负责持久化、谁负责释放 Job、谁负责操作锁。
8. 高风险流程必须保留现有 App guard、Shell 最终校验、PendingLauncherTask 和 OperationLockStore。
9. 纯逻辑拆出后补单元测试；纯 Compose 搬迁至少完成全量编译、Lint 和构建。
10. 不修改 `lukoa-tavern.sh`，除非该版本目标明确需要且行为可证明等价。

### beta28：拆对话框显示编排，不移动业务状态

从 `LukoaLauncherScreen.kt` 约 3660 到 4090 行附近的对话框调用区开始。

建议新增按域的 Host：

- `LauncherLaunchDialogHost.kt`
- `LauncherBackupDialogHost.kt`
- `LauncherProfileDialogHost.kt`
- `LauncherUpdateDialogHost.kt`
- `LauncherPermissionDialogHost.kt`

要求：

- 状态仍由 `LukoaLauncherScreen` 持有。
- Host 只决定“显示哪个对话框”和参数转发，不执行业务 IO。
- 用小型域参数对象聚合相关状态和回调；不要制造一个拥有几十个字段的总 Host 参数。
- 这一版不迁移 Job、请求 token、操作锁和 PendingTask。

### beta29：拆日志、状态归并和操作锁协调

优先处理这些本地函数：

- `extractTermuxDisplayContent`
- `update`
- `syncTermuxResult`
- `isTransientStatus`
- `beginBusy` / `releaseBusy`
- `runGuarded`
- `runPendingGuardedCommand`
- `updateTermuxLogOnly`
- 日志清除和导出入口

推荐拆成：

- `LauncherResultReducer.kt`：纯结果归并和状态推断输入输出。
- `LauncherOperationCoordinator.kt`：操作锁、超时和 PendingTask 协调。
- `LauncherLogCoordinator.kt`：日志刷新、清除、导出和归档调用。

先把可测试的状态归并做成纯 Kotlin；不要让 coordinator 同时拥有整个页面状态。

### beta30：拆备份业务域

这是最大的单一业务块，建议单独一个版本，必要时拆成 beta30 和 beta31。

范围：

- 自动备份设置。
- 备份列表刷新和本地列表读取。
- 手动备份。
- 恢复预览请求、Job 和 token。
- 恢复、复制、导出、导入、重命名、删除。
- 存储权限重试路径。

建议：

- `LauncherBackupUiState.kt`：只放备份域 UI 状态。
- `LauncherBackupCoordinator.kt`：只编排现有 backup service、codec、guard 和 PendingTask。
- 保留现有 `BackupRestorePreviewRequestCoordinator`、`BackupHistoryReducer`、`BackupLibraryFiles` 等模块，不重复实现。
- 恢复和迁移仍必须保留持久化操作锁和未完成任务恢复。

### beta31：拆实例、路径和镜像设置域

范围：

- 当前 profile 切换。
- 新增和删除 profile。
- 传统目录、托管目录和自定义目录迁移。
- 目录候选识别和选择。
- Git/npm/Termux 镜像设置。
- 路径、端口和镜像输入状态。

建议：

- `LauncherProfileCoordinator.kt`
- `LauncherPathSettingsState.kt`
- `LauncherMirrorSettingsState.kt`

不要改变默认 `~/SillyTavern`，不要自动修改用户路径，不要弱化迁移和删除确认。

### beta32：拆安装、更新和回退域

范围：

- 当前版本读取和官方版本刷新。
- 安装目标选择和安装前检查。
- apt 配置策略选择。
- 更新、回退和自动安全备份串联。
- 版本操作确认。

建议：

- `LauncherTavernVersionCoordinator.kt`
- `LauncherInstallCoordinator.kt`

必须复用：

- `TavernInstallCommandCodec`
- `TavernVersionCommandCodec`
- `TavernVersionActionGuards`
- `PendingLauncherTaskSupport`
- `TermuxCommandTimeoutPolicy`

不要在这个 UI 拆分版本顺手迁移 install/update/rollback 的 Shell 实现。

### beta33：拆启动、停止、体检和权限域

范围：

- Termux 是否安装和 RUN_COMMAND 权限。
- 后台运行权限和首次启动引导。
- 启动前预检、快速启动缓存、启动、返回酒馆。
- 停止和强制清理。
- 一键体检和快速修复。
- Termux 环境准备。

建议：

- `LauncherLaunchCoordinator.kt`
- `LauncherPermissionCoordinator.kt`
- `LauncherHealthCoordinator.kt`

启动、停止和强制清理属于高风险区，拆分时必须保持现有 token、防重复点击、运行状态推断和确认逻辑。

### beta34：收尾主页面

目标不是追求最少行数，而是让 `LukoaLauncherScreen` 最终只负责：

1. 创建和连接各业务域 state holder/coordinator。
2. 生命周期 Effect 和页面级跨域协调。
3. Pager、Header、Navigation 和 section 组装。
4. 少量真正跨域的事件路由。

建议目标：将主页面稳定缩减到约 800 至 1400 行。不要为了达到数字目标制造过度抽象。

## 8. 每版拆分时的测试策略

每个版本至少：

1. 先运行相关纯 Kotlin 单测。
2. 运行 `:app:compileDebugKotlin`。
3. 运行 `testDebugUnitTest`。
4. 运行 `:app:lintDebug`。
5. 运行 `:app:assembleDebug`。
6. 因为每版都发测试版，再运行 `:app:assembleRelease`。

完整命令：

```powershell
.\gradlew.bat --no-daemon :app:compileDebugKotlin testDebugUnitTest :app:lintDebug :app:assembleDebug :app:assembleRelease
```

UI 机械搬迁还要人工核对：

- 所有对话框标题、按钮、危险色和关闭行为不变。
- Back/点击外部区域的 dismiss 行为不变。
- 对话框互斥关系不变。
- Pager 滑动锁和 BusyPanel 不受影响。
- 备份恢复、迁移、删除、更新和回退确认没有绕过。
- App 重建后操作锁和 PendingTask 仍能恢复。

## 9. 发版规则

- 下一版本建议从 `0.9.1-beta27 / versionCode 200` 开始。
- 每完成一个清晰拆分块，发布一个 prerelease。
- 更新 `app/build.gradle.kts` 和 `CHANGELOG.md`。
- Release 公告不要写“把某文件拆成了几个文件”。
- 对外文案应写成用户收益，例如：
  - 优化页面和对话框的稳定性。
  - 降低复杂操作时界面状态互相影响的风险。
  - 改善后续功能更新和问题修复的一致性。
- 不提交 `.codex-tmp/`。
- 发布后核验提交、标签、prerelease、APK 资产、版本号和 SHA-256。
