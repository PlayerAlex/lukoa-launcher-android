## 4. 当前最重要的未完成风险

这部分很关键，下一轮不要忽略。

### 4.1 现在只是“实例配置底座”，还不是“真正完整的双实例执行层”

虽然 App 侧已经有实例和端口概念了，但 Termux 侧仍然有一个核心问题还没拆开：

- `~/.local/state/lukoa-launcher/` 下面的这些状态文件还是共用的：
  - `pid`
  - `status.json`
  - `tavern.log`
  - `last-tavern-update-commit`
  - 其他运行态缓存文件

这意味着：

- 现在可以切换“当前实例”
- 但如果想做真正可靠的双实例并行运行，还必须继续把运行状态文件按实例隔离

否则会出现：

- A 实例覆盖 B 实例的状态
- 体检读到的是另一个实例的 PID / 日志
- 停止、启动、版本回退和日志同步互相串状态

### 4.2 当前“删除实例”还没有二次确认

这轮先做的是最小可用入口，还没补危险操作确认框。

下一轮如果继续打磨实例管理，建议优先补：

- 删除实例确认
- 当前实例切换后的解释文案
- 多实例下的体检提示

### 4.3 还没做真机双实例行为验证

当前通过的是：

- 编译
- 单元测试
- lint
- debug APK 构建

还没验证的是：

- 真机上两个实例分别启动时的行为
- 两个实例是否会互相干扰
- 分别对应不同端口时的打开网页行为

## 7. 下一轮最建议的开发顺序

### 方案 A：继续把双实例底层做完整

这是我当前更推荐的顺序。

优先做：

1. 把 Termux 运行状态文件按实例隔离
2. 让 PID / status / log / rollback 等状态和实例绑定
3. 再补实例删除确认、实例改名、实例切换提示

原因：

- 现在 App 侧实例概念已经有了
- 但执行层还没真正“实例化”
- 如果这一步不先补，后面端口清理和保活功能都会建立在不稳定状态上

### 方案 B：基于现有底座开始做端口工具

如果下一轮用户更关心端口，也可以直接接这个目标，但要记住：

- 端口读取 / 修改 / 清理必须始终围绕“当前激活实例”执行
- 不要再写回“默认就是 8000”的单实例假设
- “强制端口清理”仍然只能是最后一步，不能默认执行

建议拆分文件方向：

- `TavernPortCommandSupport.kt`
- `TavernPortParser.kt`
