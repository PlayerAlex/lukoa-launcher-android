真正风险不是单纯行数多，而是：

1. 主线程里仍有 SharedPreferences JSON 解析、文件查询和文件写入。
2. Termux 结果通过轮询磁盘等待，长任务会持续造成主线程负担。
3. Kotlin 和 assets 中同时维护两套 Shell 实现。
4. `LukoaLauncherScreen.kt` 同时管理 100 个 Compose 状态、152 个本地函数和约 43 个外部回调。
5. 部分危险操作只依赖 Shell 最后拦截，App 侧没有提前防呆。
6. 命令名、参数、超时和结果文案分散在多个文件，容易改一处漏一处。

## 4. 下一版最优先任务：beta23 Termux 结果链路

这是当前最值得优先修的部分，不建议先做大规模 UI 重构。

### 4.1 当前问题

`TavernController.waitForResult()` 每 500ms 调用一次 `TermuxResultStore.recent()`。

长任务轮询次数：

- 安装最多 1800 次，约 15 分钟。
- 更新、回退、备份、恢复最多 1200 次，约 10 分钟。
- 每次轮询都会读取 SharedPreferences，并解析整段 JSON 历史记录。
- 这些轮询从主线程 CoroutineScope 启动，`delay()` 后仍会回到主线程执行解析。

关键文件：

- `app/src/main/java/moe/lukoa/launcher/TavernController.kt`
- `app/src/main/java/moe/lukoa/launcher/TermuxResultStore.kt`
- `app/src/main/java/moe/lukoa/launcher/TermuxResultReceiver.kt`
- `app/src/main/java/moe/lukoa/launcher/TermuxResultService.kt`

### 4.2 推荐实现

新增独立的结果仓库，例如：

- `TermuxResultRepository.kt`
- `TermuxCommandTimeoutPolicy.kt`

推荐职责：

1. BroadcastReceiver / Service 收到结果后，将结果推送到 `SharedFlow` 或 `StateFlow`。
2. `TavernController` 使用 `withTimeout` + `first { matchesCurrentCommand(...) }` 等待匹配结果。
3. SharedPreferences 只承担冷启动恢复和诊断历史，不再作为实时消息队列。
4. 如果必须保留轮询兜底，必须放到 `Dispatchers.IO`，并降低频率，不能每 500ms 在主线程解析 JSON。
5. 命令超时必须集中配置，避免 UI 忙碌锁和控制器等待时间不一致。

### 4.3 Receiver / Service 风险

当前 `TermuxResultReceiver.onReceive()` 和 `TermuxResultService.onStartCommand()` 会同步执行：

- `TermuxResultStore.save()`
- `AutoBackupRetentionManager.maybePruneAfterTermuxResult()`

自动备份保留清理可能包含：

- MediaStore 查询。
- 备份目录扫描。
- 文件排序和删除。
- 日志写入。

这些操作不应直接运行在广播接收主线程。

推荐：

- Receiver 使用 `goAsync()`，在 IO 调度器完成最小持久化。
- 自动备份清理改为一次性 WorkManager 任务。
- Receiver 只负责解析、投递、安排后台任务，不负责扫描整个备份库。

### 4.4 结果存储需要一起收口

`TermuxResultParser.buildRawDump()` 会把整个 Bundle 再展开一次，通常与 `stdout` 重复。

`TermuxResultStore` 又同时保存：

- `stdout`
- `stderr`
- `raw`
- 24 条历史记录 JSON

建议：

- `raw` 只保留未知元数据，不重复保存 stdout/stderr。
- 对历史结果设置明确字符上限。
- 最新完整结果和诊断历史分开存储。
- 补 `TermuxResultStoreTest`、`TermuxResultRepositoryTest` 和命令匹配测试。

## 5. beta24 建议：备份恢复后台化和防呆

### 5.1 当前主线程 I/O

`openApplyBackupPreview()` 会直接调用 `BackupRestorePreviewResolver.resolve()`。

该流程可能：

- 查询 MediaStore。
- 扫描备份库。
- 打开文件读取 gzip 头。

`applySelectedBackup()` 还会同步调用 `BackupLibraryFiles.canReadLibrarySource()`。

这些都应放到 `Dispatchers.IO`。

关键文件：

- `app/src/main/java/moe/lukoa/launcher/LukoaLauncherScreen.kt`
- `app/src/main/java/moe/lukoa/launcher/BackupRestorePreview.kt`
- `app/src/main/java/moe/lukoa/launcher/BackupLibraryFiles.kt`

### 5.2 推荐模块

新增：

- `BackupRestorePreflight.kt`
- `BackupCoordinator.kt`

建议状态：

- `Idle`
- `Checking`
- `Ready(preview)`
- `Blocked(reason)`
- `Failed(message)`

对话框生成期间显示“正在检查备份”，未检查完成前不能点击应用。

### 5.3 必补防呆

应用备份前 App 侧必须检查：

- 当前实例是否正在运行。
- 当前实例是否正在启动。
- 是否存在其他挂起任务。
- 备份是否可读、非空、gzip 格式有效。
- 恢复目标实例、目录和端口是否仍与打开预览时一致。
- 用户打开预览后切换实例时，旧预览必须立即失效。

当前 Shell 已经有路径穿越、备份根目录和运行状态检查，不要删除 Shell 兜底；App 防呆是第一层，Shell 校验仍是最终安全层。

### 5.4 权限重试问题

`retryApplyAfterTermuxStoragePermission()` 当前会先把 `termuxStoragePermissionBlocked` 设为 false，再直接重试恢复。

用户如果拒绝授权，App 状态会短暂撒谎。

应改为：

1. 重新执行真实权限检测。
2. 确认可读后才清除 blocked 状态。
3. 仍不可读时保持弹窗或给出明确失败说明。

## 6. beta25 建议：结构化命令、统一超时和操作锁

当前仍使用类似：

- `tavern-restore::路径`
- `tavern-update::编码参数`

这种字符串协议。

建议新增：

- `sealed interface LauncherCommand`
- `data class LauncherCommandSpec`
- `LauncherOperationCoordinator`

每个命令统一包含：

- 命令类型。
- 结构化参数。
- 用户显示名称。
- 超时时间。
- 是否为危险操作。
- 是否要求酒馆停止。
- 是否需要持久化为 PendingLauncherTask。
- 成功后的刷新目标。

不要让超时继续分别散落在：

- `TavernController.attemptsForCommand()`
- `LukoaLauncherScreen.beginBusy()`
- 各个调用点的 `timeoutMs`

### 6.1 持久化操作锁

`OperationLockStore` 会保存锁，但 App 重建后 `busyLabel` 仍从 null 开始。

下一轮应：

- 启动时恢复未过期的操作锁。
- 将操作锁与 `PendingLauncherTask` 合并为一个统一状态机。
- 未确认旧任务结束前，禁止再次发出冲突的安装、更新、回退、备份、恢复、迁移命令。

## 7. beta26 建议：TermuxCommandRunner 去双实现

### 7.1 当前问题

`TermuxCommandRunner.kt` 同时包含：

- assets 脚本读取和安装逻辑。
- Kotlin 内嵌的约 676 行共享 Shell 前置代码。
- 安装、更新、回退等独立 Shell 命令生成器。
- 已经没有调用者的旧停止、版本、官方版本命令生成函数。

这意味着修改路径规则、状态目录、PID、日志或候选目录时，很容易只改一套。

### 7.2 推荐边界

拆成：

- `TermuxCommandDispatcher.kt`：Android Intent、权限、PendingIntent、发送命令。
- `TermuxScriptInstaller.kt`：读取 assets、计算哈希、安装/升级脚本。
- `TermuxCommandFactory.kt`：结构化参数转换，不承载业务 Shell。
- `TermuxResultRepository.kt`：结果接收、流式通知和有限持久化。

原则：

- `app/src/main/assets/lukoa-tavern.sh` 是酒馆业务 Shell 的唯一实现。
- Kotlin 不再复制完整的路径识别、状态文件、PID、备份和 Git 操作逻辑。
- 普通命令直接执行已安装脚本。
- 脚本缺失或哈希不一致时才重装。
- 删除旧代码前先为命令路由、参数编码和脚本版本检查补测试。

## 8. 后续模块化拆分：LukoaLauncherScreen

不要一次性重写整个 4452 行文件。

建议按以下顺序渐进迁移：

1. `LauncherOperationCoordinator`
   - 忙碌锁。
   - PendingLauncherTask。
   - 超时和恢复。
2. `LauncherRuntimeStateReducer`
   - 状态、摘要、运行状态推断。
   - Termux 输出归并。
   - 日志同步。
3. `BackupCoordinator`
   - 备份列表、导入、导出、重命名、复制、删除、恢复。
4. `VersionCoordinator`
   - 当前版本、官方列表、安装、更新、回退。
5. `LauncherDialogState`
   - 用一个密封状态替代几十个 `showXxxDialog` Boolean。

可以最终落到一个根 `LauncherViewModel`，但应先提取纯逻辑和协调器，再迁移 Compose 状态，避免一次性行为回归。

## 10. 其他防呆建议

### 10.1 更新和回退

`TavernVersionActionGuards` 当前会检查：

- 是否安装。
- 是否读取当前版本。
- 是否有本地改动。
- Git 源和官方版本列表是否匹配。

但没有 App 侧检查酒馆是否正在运行或启动。

建议在打开确认框前直接阻止，并提供“先停止当前实例”的明确提示。Shell 的 `ensure_tavern_mutation_ready` 仍保留为最终兜底。

### 10.2 高风险确认强度

已有普通确认框的动作：

- 应用备份。
- 删除备份。
- 删除实例。
- 迁移目录。
- 停止。
- 强制清理。
- 更新和回退。

建议仅对真正会删除目录或覆盖数据的场景提高确认强度：

- 恢复备份：要求输入“应用”。
- 删除带托管目录的实例：要求输入实例名或“删除”。
- 强制清理：如果检测目标不明确，要求输入“强制清理”。

不要给所有普通操作都加文字确认，避免用户形成无脑输入习惯。

### 10.3 Compose 重组性能

以下初始化解析目前没有 remember，会在重组时重复执行：

- 初始运行状态推断。
- 当前版本解析。
- 官方版本缓存解析。
- 问题分析 `TavernIssueAnalyzer.analyze()`。

这些应使用 `remember(...)` 或移入状态协调器/ViewModel。

### 10.4 日志写入

beta22 只修复了导出时整文件读取。

仍需处理：

- `RuntimeLogArchive.appendText()` 同步写文件。
- 本地日志归档没有大小上限。

建议：

- 日志写入放单线程 IO 队列。
- 每类日志设置高水位和保留水位，例如超过 4MB 后裁剪到最近 2MB。
- 裁剪不能放在 Compose 主线程。
